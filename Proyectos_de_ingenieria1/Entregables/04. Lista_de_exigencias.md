    </td>
    <td>
      <p>B.B., S.S.</p>
      <p>M.T., J.P.</p>
    </td>
  </tr>

  <!-- Fila 2: Geometría  -->
  <tr>
    <td><b>31/03/26</b></td>
    <td>E</td>
    <td>
      <b>Geometría:</b> El dispositivo IoT de validación en campo debe ser compacto, resistente a condiciones altoandinas (viento, lluvia, radiación UV, granizo). <b>Permitirá montaje en postes, cercos o estructuras agrícolas sin herramientas especializadas.</b> La carcasa será inoxidable y membrana respiradora para evitar condensación interna.</b>
    </td>
    <td>B.B.</td>
  </tr>

  <!-- Fila 3: Señales  -->
  <tr>
    <td><b>31/03/26</b></td>
    <td>E</td>
    <td>
      <p><b>Señales (Información)</b>:</p>
      <p><i>Señales de entrada:</i></p>
      <ul>
        <li>Sensor de temperatura para detectar umbrales de helada (<0 °C) y estrés térmico.</li>
        <li>Sensor de humedad relativa, para identificar sequía o exceso hídrico.</li>
        <li>Sensor de humedad de suelo capacitivo, para validar estrés hídrico a nivel radicular.</li>
        <li>Módulo GPS , para geolocalización hiperlocal de alertas.</li>
        <li><b>Indicadores de estado de carga, conexión y alertas.</b></li>
      </ul>
      <p><i>Señales de salida:</i></p>
      <ul>
        <li>Alertas push/SMS multicanal (español/quechua) con recomendaciones agronómicas contextualizadas.</li>
        <li>Transmisión de datos validados a dashboard web/móvil para visualización en tiempo real.</li>
        <li><b>Exportación de datos históricos y predicciones (CSV, JSON, PDF, Excel).</b></li>
      </ul>
    </td>
    <td>B.B., S.S.</td>
  </tr>

  <!-- Fila 4: Control  -->
  <tr>
    <td><b>31/03/26</b></td>
    <td>E</td>
    <td>
      <b>Control:</b> El sistema incorporará lógica de umbrales configurables por cultivo y zona geográfica, con estrategias de alerta escalonadas (advertencia → alerta → emergencia). Las recomendaciones se basarán en buenas prácticas agronómicas validadas por INIA y CIP [11]. El diseño priorizará sostenibilidad: bajo consumo, modo sleep programable. <b>Incluye extrapolación temporal (series históricas 5-10 años) y espacial (TEDs: Dominios de Extrapolación Tecnológica).</b>
    </td>
    <td>B.B.</td>
  </tr>

  <!-- Fila 5: Electrónico (hardware)  -->
  <tr>
    <td><b>31/03/26</b></td>
    <td>E</td>
    <td>
      <b>Electrónico (hardware):</b> Se utilizará microcontrolador integrado con sensores calibrados para condiciones andinas. Se alimentará mediante batería recargable con panel solar incluido. <b>El circuito incluirá protección contra sobretensión, polaridad inversa y ruido electromagnético.</b> La carcasa tendrá grado de protección estándar para resistencia a polvo, lluvia e inmersión temporal.</b>
    </td>
    <td>B.B.</td>
  </tr>

  <!-- Fila 6: Software -->
  <tr>
    <td><b>31/03/26</b></td>
    <td>E</td>
    <td>
      <b>Software:</b> Se desarrollará una aplicación móvil/web bajo principios de diseño centrado en el usuario (ISO 9241-210:2019) [2], con interfaz intuitiva para usuarios de baja alfabetización digital. El dashboard mostrará: (1) mapa interactivo con alertas hiperlocalizadas <b>y zonas monitoreadas por ID</b>, (2) histórico de predicciones vs. mediciones en campo, (3) recomendaciones agronómicas ilustradas, <b>(4) predicciones a 10 meses con gráficos de tendencia e intervalos de confianza.</b> La arquitectura será offline-first: funcionará sin conectividad, sincronizando datos cuando haya red. Las alertas se generarán mediante comparación entre predicciones satelitales y mediciones locales. <b>Exportación multi-formato con un click.</b>
    </td>
    <td>B.B., J.P.</td>
  </tr>

  <!-- Fila 7: Comunicación -->
  <tr>
    <td><b>31/03/26</b></td>
    <td>D</td>
    <td>
      <b>Comunicación:</b> El sistema deberá integrar conectividad <b>comunicación sensor-app</b> (alcance ≥10 m) y WiFi/celular <b>(solo gateway)</b> para sincronización con la nube. Se utilizará protocolo MQTT (ISO/IEC 20922:2016) [1] para transmisión eficiente de datos IoT. <b>Arquitectura distribuida.</b> Se garantizará interoperabilidad con APIs satelitales y plataformas estatales.
    </td>
    <td>B.B., S.S.</td>
  </tr>

  <!-- Fila 8: Ergonomía  -->
  <tr>
    <td><b>31/03/26</b></td>
    <td>E</td>
    <td>
      <p><b>Ergonomía:</b> La interfaz de usuario cumplirá con principios de accesibilidad (ISO 9241-210:2019) [2]: iconografía intuitiva, texto en español/quechua, contraste alto para visibilidad bajo luz solar, y navegación con máximo 3 toques para acceder a alertas críticas. Los indicadores visuales usarán codificación por color (verde=seguro, amarillo=precaución, rojo=emergencia) conforme a ISO 9241-112:2025 [3]. El dispositivo físico incluirá <b>4 LEDs indicadores de estado (carga, batería, conexión, alerta)</b> y botón físico de emergencia para activar alerta manual.</p>
    </td>
    <td>J.P., B.B.</td>
  </tr>

  <tr>

(2) La calibración de sensores contra patrones certificados (error <5%). (3) Pruebas de campo en zonas piloto (Lomas de Pachacamac o simuladores) para validar precisión predictiva. (4) Las pruebas validarán el desempeño del sistema en condiciones reales y controladas.</p>
